# Test Writing Rules

## 必須コメントルール

すべてのテストファイルとテストケースに、以下の形式でコメントを記載すること。
コメントは日本語で記述する。

### 1. ファイルレベルコメント（必須）

各テストファイルの先頭に JSDoc 形式で以下の3項目を記載する:

```typescript
/**
 * Unit tests for <テスト対象>
 *
 * What: <このテストファイルが検証する内容の要約>
 *
 * Why:  <なぜこのテストが必要なのか、ビジネス上の理由>
 *
 * Risk if failing:
 *   - <テストが失敗した場合に起こりうる具体的な影響 1>
 *   - <テストが失敗した場合に起こりうる具体的な影響 2>
 */
```

### 2. テストケースレベルコメント（必須）

各 `it()` / `test()` ブロックの直前に JSDoc 形式で以下の3項目を記載する:

```typescript
/**
 * What: <このテストが検証する具体的な振る舞い>
 * Why:  <なぜこの振る舞いの検証が重要なのか>
 * Risk: <このテストが失敗した場合の具体的なリスク>
 */
it('test description', async () => {
  // ...
})
```

### 3. ルール

- **What**: 「何を検証するか」を1-2文で簡潔に記述する
- **Why**: 「なぜその検証が必要か」をビジネス/ユーザー影響の観点で記述する
- **Risk**: 「テスト失敗時に何が起こるか」を具体的なシナリオで記述する
- `it.each()` など複数ケースをまとめるテストでは、グループ全体に1つのコメントを付ける
- コメントは既存テストのリファクタリング時にも削除してはならない
- Prettier/ESLint のフォーマット適用後もコメントが保持されることを確認する

### 4. Python テストの場合

Python テストでは docstring 形式で同等の情報を記載する:

```python
def test_example():
    """
    What: <検証内容>
    Why:  <必要性>
    Risk: <失敗時のリスク>
    """
```
